# SyncroBrain 平台愿景 (v1.1)

> **品牌**：[SyncroBrain](https://syncrobrain.com) · **组织**：[github.com/syncrobrain](https://github.com/syncrobrain)（原 LuminaryIoTChain）  
> **定位**：开源、轻量化、AI 驱动的 IoT PaaS — 深耕 **B 端垂直行业**与**白牌出海**，以更低成本与更强 AI 能力服务硬件厂商与产业集群。

## 1. 我们要解决什么

| 痛点 | 解决方案 |
|------|----------|
| **硬件厂商缺乏 IoT 能力** | 提供即插即用的 IoT 平台、App 与云服务，无需自建底层系统 |
| **生态碎片化** | 统一协议（MQTT）与云平台，实现不同品牌设备互联互通 |
| **国际化难度高** | 多语言、多区域云服务架构，帮助厂商快速海外布局 |
| **大厂不愿做的长尾行业** | 行业协议解调（Decoder）+ 专属 BI 看板，把 Modbus/BACnet/OPC-UA 转成业务指标 |
| **数据合规与主权诉求** | 可私有化部署，数据不出园区/国境；满足 GDPR、NIS2、信创与《数据安全法》 |
| **白牌厂出海又不想被平台绑定** | 开源适配层 + 贴牌独立 App，几万块搭建自有「品牌云」 |

## 2. 大平台的软肋，小平台的生路

涂鸦、AWS 等大平台无法覆盖所有**非标**与**长尾**需求。SyncroBrain 不与之正面争夺消费级通用 IoT，而是切入大厂不愿深耕的缝隙：

| 机会 | 我们的做法 |
|------|------------|
| **垂直行业解调与 BI** | 畜牧业智能养殖、高压电网无损检测、实验室超低温冰箱群监控、危化品仓储等 — 年产值数百万的细分行业，大厂不会定制行业 Decoder 与专属业务看板；我们懂行业协议，能把遥测翻译成客户看得懂的 KPI |
| **合规与本地化** | 中小企业、园区、政府项目日益拒绝将数据上传至涂鸦云或 AWS 公有云；我们提供本地化部署、全天候售后、White-label 专属 App |
| **私有化成本鸿沟** | 大厂私有化动辄数十万起步；我们以开源栈帮客户**数万级**搞定 |
| **白牌出海** | 国内中小硬件白牌厂想出海，但涂鸦对接成本/门槛高，或不愿被平台控制供应链；联合乐鑫 ESP32、国产 T5 等芯片生态，提供极简开源适配层，帮其搭建自有微型品牌云 |

> **客户画像原则**：不服务「所有人」，只服务**某一代工链**或**某一产业集群**（例如：顺德某小家电集群的共享租赁平台，或某特定农业基地的 IoT SaaS）。详见 [§7 初期红线](#7-初期红线规避小平台倒闭的死穴)。

## 3. 与涂鸦（Tuya）的差异

| 维度 | 涂鸦 | SyncroBrain |
|------|------|-------------|
| 商业模式 | 闭源 SaaS，按设备/流量收费 | **开源可私有化**，部署与许可成本更低 |
| 能力边界 | 以设备管理、场景联动为主 | **AI 驱动**：推理、Agent、链上交易、数据洞察 |
| 开发者生态 | 涂鸦开发者平台 | **BlockyEdu** AI 辅助接入与实验课程 |
| 数据可视化 | 内置面板 | **DataLuminary DataTalk** 专业 BI 大屏 |
| 远程运维 | 有限 | **VistaRemote** WebRTC 远程桌面 |
| 变现 | 平台抽成 | 设备可接入 **DoerFlow** 任务网络，设备侧亦可参与链上收益 |

## 4. 三大差异化能力

### 4.1 接入快 — BlockyEdu 赋能工程师

- BlockyEdu 提供 ESPHome / MQTT / ThingsBoard 实验课程与 AI 辅导
- 厂商工程师通过 code-app-web 快速完成固件模板与协议对接
- 降低从「硬件样品」到「云端在线」的时间

### 4.2 AI 服务 — 不止设备管理

- DataLuminary：时序分析、异常检测、可视化大屏
- DoerFlow：视觉识别、控制 Agent 订阅与链上结算
- 设备遥测 → AI 推理 → 规则/automation 闭环

### 4.3 区块链交易平台 — 设备也能赚钱

- 设备可将算力/数据/Agent 能力注册到 DoerFlow 网络
- 用户购买 AI 服务；设备拥有者可获得链上分成
- LuminaryIoTChain 编排身份（Logto `sub`）与设备 ID 的映射

## 5. 核心链路

```text
设备（端） ──MQTT──► 物联网平台（大脑） ──API──► 终端 App（展示）
                         │
                         ├── ThingsBoard CE（设备影子、规则、多租户）
                         ├── DataLuminary DataTalk（图表 / 大屏）
                         ├── DoerFlow（AI 服务 / 链上交易）
                         ├── BlockyEdu（开发者接入）
                         ├── VistaRemote（远程桌面运维）
                         └── VistaCast（摄像头 AI，规划）
```

## 6. 开源与收费策略（原则）

- **核心平台**（ThingsBoard CE、EMQX OSS、LuminaryIoTChain 编排层）— Apache / MIT 可私有化
- **增值能力**（托管云、AI 推理配额、链上 Gas 代付、企业 SLA）— 低于涂鸦同等量级报价
- **不锁定厂商**：标准 MQTT + REST API，可随时迁出

## 7. 初期红线：规避小平台倒闭的死穴

项目初期必须遵守以下三条，避免重蹈小 IoT 平台覆辙：

### 死穴一：陷入高额云基础设施成本（带宽与高并发）

**破局**：不做高频高并发的 C 端消费级设备（百万级同时在线的智能灯泡、摄像头等）。只做 **B 端高客单价、低频次** 的数据采集 — 例如每小时上报一次的环境监测站、冷链物流车、工业传感器节点。

### 死穴二：试图自己造所有的轮子

**破局**：不从零编写底层 MQTT 代理或时序数据库。依托成熟开源基础设施，**只负责用 React / NestJS 缝合核心业务**：

| 能力 | 选型 | 我们负责 |
|------|------|----------|
| 消息连接 | **EMQX** (OSS) | 认证、桥接、租户隔离配置 |
| 设备引擎与规则 | **ThingsBoard CE** | 行业 Decoder、多租户编排 |
| 时序存储 | **ClickHouse** / InfluxDB / TimescaleDB | 行业指标聚合与查询 API |
| 拓荒逻辑 | **Node-RED** / TB 规则链 | 快速对接非标协议 |
| 业务与 BI | **iot-gateway** + **DataTalk** | 定制看板、AI 数据分析 |

### 死穴三：产品没有明确的客户画像

**破局**：不服务「所有人」。每次立项必须能回答：**服务哪一代工链、哪一产业集群**。

| ✅ 可做 | ❌ 不做 |
|---------|---------|
| 顺德某小家电集群的共享租赁平台 | 通用智能家居 App |
| 某特定农业基地的 IoT SaaS | 面向所有人的设备管理平台 |
| 某危化品仓储园区的合规监控 | 消费级摄像头云存储 |

> 以上红线写入架构与里程碑评审：新功能若拉高连接频次、重复造轮子、或模糊客户边界，应被拒绝或延后。
